# Monitoring Stack — Terraform + Ansible

Grafana, Prometheus, Pushgateway, Loki, and Promtail deployed across two EC2 instances in AWS. Infrastructure provisioned by Terraform, services configured by Ansible running Docker containers.

## Architecture

```
Internet
   │
   ▼
[Internet Gateway]
   │
   ├── Public Subnet (10.0.1.0/24)
   │      └── monitoring-public  [Elastic IP]
   │             ├── Grafana      :3000
   │             └── Pushgateway  :9091
   │
   └── Private Subnet (10.0.2.0/24)
          └── monitoring-private  [no public IP]
                 ├── Prometheus   :9090
                 ├── Loki         :3100
                 └── Promtail     :9080
                          │
                 [NAT Gateway] ← outbound internet for Docker image pulls
```

Prometheus, Loki, and Promtail are only reachable within the VPC. Grafana datasources and Prometheus scrape targets are templated with actual private IPs at deploy time — no hostnames hardcoded.

Ansible discovers hosts at runtime using the AWS EC2 dynamic inventory plugin, filtering by `Project=monitoring` tag. No static inventory file.

SSH keys are generated by Terraform and stored in SSM Parameter Store (SecureString). The Ansible workflow fetches the key at runtime — no `.pem` files in the repo or managed manually.

## Prerequisites

- AWS account with an IAM user that has permissions for: EC2, VPC, SSM Parameter Store, S3, KMS
- An S3 bucket for Terraform remote state (versioning + KMS encryption recommended)
- A GitHub repository with Actions enabled

## Setup

### 1. Add GitHub Secrets

| Secret | Description |
|---|---|
| `AWS_ACCESS_KEY_ID` | IAM user access key |
| `AWS_SECRET_ACCESS_KEY` | IAM user secret key |
| `AWS_DEFAULT_REGION` | Region to deploy into, e.g. `us-east-1` |
| `TF_STATE_BUCKET` | S3 bucket name for Terraform remote state |

### 2. Configure Terraform variables (optional)

```bash
cp terraform/terraform.tfvars.example terraform/terraform.tfvars
```

Edit `terraform.tfvars` to change the region or instance type. The file is gitignored.

### 3. Deploy infrastructure — Terraform

Go to **Actions → Terraform → Run workflow** and select `plan` first to review what Terraform will create. Once happy, run it again with `apply`.

The apply run:
- Creates the VPC, subnets, NAT Gateway, and security groups
- Generates an RSA 4096 SSH keypair
- Registers the public key with EC2
- Stores the private key in SSM Parameter Store at `/monitoring/ssh-private-key`
- Launches both EC2 instances with the appropriate tags

### 4. Deploy services — Ansible

Go to **Actions → Ansible Deploy → Run workflow** in GitHub, then click **Run workflow**.

The workflow:
1. Installs `boto3`, `ansible`, and the `amazon.aws` + `community.docker` collections
2. Fetches the SSH private key from SSM Parameter Store
3. Runs `ansible-playbook playbooks/site.yml`, which:
   - Installs Docker on both hosts via `dnf`, enables it as a systemd service
   - Deploys Prometheus, Loki, and Promtail on the private host
   - Deploys Grafana and Pushgateway on the public host
   - All containers run with `restart_policy: always`

### 5. Access the stack

Get the Elastic IP from Terraform outputs:

```bash
cd terraform && terraform output public_ip
```

Or check the apply step output in the GitHub Actions run.

| Service | URL | Credentials |
|---|---|---|
| Grafana | `http://<public_ip>:3000` | admin / admin |
| Pushgateway | `http://<public_ip>:9091` | — |

## Local development

Requires: AWS CLI configured, `terraform`, `ansible`, `boto3` installed.

```bash
# Deploy everything
export TF_STATE_BUCKET=your-bucket-name
export AWS_DEFAULT_REGION=us-east-1
make all

# Or step by step
make init
make apply
make fetch-key   # downloads SSH key from SSM to ansible/.keys/monitoring.pem
make deploy      # runs ansible-playbook
```

## Testing the Pushgateway

Push a test metric:

```bash
export PUSHGATEWAY_IP=<your_ip>

echo "hello_world 1" | curl --data-binary @- http://$PUSHGATEWAY_IP:9091/metrics/job/hello_world
```

Verify it was received by the Pushgateway:

```bash
curl -s http://$PUSHGATEWAY_IP:9091/metrics | grep hello_world
```

From there the flow is automatic - Prometheus scrapes the Pushgateway every 15 seconds (default), so within a few seconds you can query `hello_world` in Prometheus at `http://<PROMETHEUS_IP>:9090` and it will return the value you pushed.

In Grafana, open **Explore**, select the Prometheus datasource, and query `hello_world` - you should see the metric plotted on the graph. This confirms the full pipeline: Pushgateway -> Prometheus -> Grafana is working end to end.

## Container restart behaviour

Each container runs with `--restart=always` (via `restart_policy: always` in the Ansible docker_container module). Docker itself is configured as a systemd service that starts on instance boot.

Result:
- **Container crash** → Docker restarts it automatically within seconds
- **Host reboot** → systemd starts Docker, Docker restarts all `always` containers

This does not cover instance termination. True HA for this stack requires Thanos or Mimir (Prometheus), distributed Loki, and a replicated push endpoint — a separate architectural undertaking.

## Teardown

```bash
make destroy
```

Terminates both EC2 instances, removes the VPC and all networking, deletes the SSH key from SSM Parameter Store.
